# Как выкатить изменения на прод (подробно, без «магии»)

Эта инструкция для того, кто **не программирует каждый день**. Цель: после правок в проекте сайт на сервере обновился и показывал новую версию.

---

## Словарь (один раз прочитать)

| Термин | Простыми словами |
|--------|------------------|
| **Репозиторий (repo)** | Папка проекта на GitHub — там хранится код. |
| **Коммит** | Сохранённый «снимок» кода с подписью (сообщением). |
| **Пуш (push)** | Отправка ваших коммитов с компьютера на GitHub. |
| **Прод (production)** | «Боевой» сайт, который видят пользователи. |
| **VPS / сервер** | Удалённый компьютер, где крутится сайт и база. |
| **Деплой** | Копирование свежего кода на сервер и перезапуск программ в Docker. |
| **Секреты GitHub** | Пароли и ключи, которые GitHub хранит скрыто и подставляет в автодеплой. |
| **GitHub Actions** | Автоматические сценарии на GitHub: например «при пуше в main — залить код на сервер». |

---

## Два способа деплоя (выберите один основной)

### Способ A — через GitHub (удобно, если всё настроено)

Вы делаете **push в ветку `main`**. GitHub сам копирует код на сервер и выполняет `docker compose … --build`.

**Плюсы:** не нужно помнить команды сервера.  
**Минусы:** один раз нужно правильно настроить **четыре секрета** и **SSH-ключ** на сервере. Если что-то не так — в Actions будет красный крестик.

### Способ B — вручную со своего Mac/ПК (скрипт)

Вы запускаете на своём компьютере скрипт `scripts/deploy-to-prod.sh`: он копирует файлы на сервер по SSH и там пересобирает Docker.

**Плюсы:** не зависит от секретов в GitHub.  
**Минусы:** нужен установленный `ssh`, доступ к серверу, переменные окружения (см. ниже).

---

# Часть 1. Что должно быть готово **до** любого деплоя

## 1.1. На GitHub

- Репозиторий с кодом (например `vldraven/avtovozom.com`).
- Ветка **`main`** — в неё попадает то, что должно уйти на прод.

## 1.2. На сервере (VPS)

- Установлены **Docker** и **Docker Compose** (плагин `docker compose`).
- Есть каталог проекта, чаще всего **`/opt/avtovozom`** (или тот, что вы задали в секретах).
- В корне проекта на сервере лежит файл **`.env`** с паролями и адресами (**он никогда не коммитится в Git**).
- Папка **`media/`** для фото — её деплой **не затирает** (она в исключениях).

Если `.env` на сервере нет — сайт после сборки может не подняться. Скопируйте с локального примера и заполните: в корне репозитория есть **`.env.example`**.

# Часть 2. Способ A — автодеплой через GitHub Actions

## 2.1. Как это работает (картинка в голове)

1. Вы вносите изменения в код на компьютере.  
2. Делаете **commit** и **push** в **`main`**.  
3. На GitHub запускается workflow **«Deploy production»** (файл `.github/workflows/deploy-production.yml`).  
4. Он проверяет **секреты**, копирует код на сервер (**rsync** по SSH), затем на сервере выполняет:

   `docker compose -f docker-compose.prod.yml up -d --build`

## 2.2. Обязательные секреты (без них деплой упадёт)

Откройте в браузере:

**GitHub → ваш репозиторий → Settings → Secrets and variables → Actions → вкладка Secrets**

Должны быть **ровно четыре** секрета (имена **как в таблице**, регистр важен):

| Имя секрета | Что вписать |
|-------------|-------------|
| `DEPLOY_HOST` | IP сервера (например `91.196.33.68`) или домен, если он стабильно открывается. |
| `DEPLOY_USER` | Пользователь SSH, чаще `root` или отдельный пользователь с правами на Docker. |
| `DEPLOY_PATH` | Путь к проекту на сервере **без** слэша в конце, например `/opt/avtovozom`. |
| `DEPLOY_SSH_KEY` | **Целиком** приватный SSH-ключ (многострочный текст от `-----BEGIN` до `-----END`). |

**Как добавить секрет:** кнопка **New repository secret** → в поле **Name** — имя из таблицы → в **Secret** — значение → **Add secret**.

### Частая ошибка: «нажал Run workflow, всё красное, exit code 255»

**Код 255** в таких сценариях чаще всего значит:

1. **Не заданы или пустые секреты** — в логе шага будет строка вроде  
   `Missing secrets: DEPLOY_HOST, DEPLOY_USER, DEPLOY_PATH, DEPLOY_SSH_KEY`  
   Тогда зайдите в Settings → Secrets и добавьте все четыре.

2. **Неверный ключ** — GitHub не может зайти на сервер по SSH. Проверьте:
   - в `DEPLOY_SSH_KEY` вставлен **приватный** ключ, не публичный (не файл `.pub`);
   - в конце ключа **нет лишних пробелов** и не обрезаны строки;
   - на сервере в `~/.ssh/authorized_keys` пользователя из `DEPLOY_USER` добавлен **соответствующий публичный** ключ.

3. **Неверный хост или путь** — `DEPLOY_HOST` не тот IP, `DEPLOY_PATH` на сервере не существует или нет прав.

4. **На сервере нет Docker или нет файла** `docker-compose.prod.yml` в каталоге `DEPLOY_PATH`.

**Как смотреть причину точно:**

GitHub → **Actions** → слева **Deploy production** → откройте упавший запуск → раскройте шаг **«Rsync and docker compose»** → прокрутите лог **вверх с конца** — там обычно видно: «Permission denied», «Connection refused», «Missing secrets», ошибка `docker` и т.д.

Предупреждение про **Node.js 20** внизу — это **не причина падения**, это уведомление GitHub на будущее. На работу деплоя сейчас не влияет.

## 2.3. SSH-ключ только для деплоя (делается один раз)

Кратко (подробнее в [GITHUB_AND_DOMAIN.md](GITHUB_AND_DOMAIN.md)):

1. На **своём Mac** создайте пару ключей, например:

   ```bash
   ssh-keygen -t ed25519 -f ~/.ssh/avtovozom_deploy -N "" -C "github-actions-deploy"
   ```

2. Содержимое файла **`avtovozom_deploy.pub`** (одна строка) добавьте на **сервер** в  
   `/root/.ssh/authorized_keys` (или у того пользователя, что в `DEPLOY_USER`).

3. Содержимое файла **`avtovozom_deploy`** (приватный, много строк) целиком вставьте в секрет GitHub **`DEPLOY_SSH_KEY`**.

4. Проверка с Mac:

   ```bash
   ssh -i ~/.ssh/avtovozom_deploy ВАШ_USER@ВАШ_IP
   ```

   Должно пустить **без пароля**.

## 2.4. Обычный цикл «выложить изменения» (когда секреты уже есть)

На компьютере, в папке проекта:

```bash
cd /путь/к/avtovozom.com
git status
git add -A
git commit -m "Кратко что сделали"
git push origin main
```

Дальше откройте **Actions** и дождитесь зелёной галочки у **Deploy production**.

**Запуск вручную без нового коммита:**  
Actions → **Deploy production** → справа **Run workflow** → ветка `main` → **Run workflow**.

# Часть 3. Способ B — деплой скриптом с компьютера

Подходит, если GitHub Actions по каким-то причинам не используете.

## 3.1. Один раз настроить переменные

В терминале (можно добавить в `~/.zshrc`, чтобы не вводить каждый раз):

```bash
export DEPLOY_HOST=ВАШ_IP_ИЛИ_ДОМЕН
export DEPLOY_USER=root
export DEPLOY_PATH=/opt/avtovozom
```

`DEPLOY_PATH` должен совпадать с реальным каталогом проекта на сервере.

## 3.2. SSH без пароля

Тот же ключ, что и для GitHub, или ваш личный ключ — главное, чтобы команда заходила на сервер:

```bash
ssh $DEPLOY_USER@$DEPLOY_HOST
```

## 3.3. Запуск деплоя

Из **корня репозитория** на Mac:

```bash
chmod +x scripts/deploy-to-prod.sh
./scripts/deploy-to-prod.sh
```

Скрипт:

- копирует код (**не** трогая на сервере `.env` и `media/`);
- выполняет на сервере `docker compose -f docker-compose.prod.yml up -d --build`.

Если скрипт ругается на `DEPLOY_HOST` — переменная не экспортирована, вернитесь к шагу 3.1.

---

# Часть 4. Проверка после деплоя

1. На сервере (если есть доступ по SSH):

   ```bash
   cd /opt/avtovozom
   docker compose -f docker-compose.prod.yml ps
   ```

   У сервисов статус должен быть что-то вроде **running**.

2. В браузере откройте сайт и пару внутренних страниц (каталог, одно объявление).

3. Если меняли только переменные вида **`NEXT_PUBLIC_...`**, обязательно нужна **пересборка образа web** — и скрипт, и Actions это делают через `--build`.

# Часть 5. Если ничего не получается — кому что написать

Соберите **три вещи**:

1. Скрин или текст ошибки из **GitHub Actions** (шаг **Rsync and docker compose**, последние 30–50 строк лога).  
2. Точные значения **без паролей**: `DEPLOY_HOST`, `DEPLOY_USER`, `DEPLOY_PATH` (секрет ключ не присылайте в чат — только факт «ключ добавлен / проверяли ssh с Mac»).  
3. Работает ли ручной вход: `ssh user@host`.

Разработчик по логу обычно за 5–10 минут видит: секреты, ключ, путь, Docker или сеть.
